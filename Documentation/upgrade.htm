Before you start

Please check the Prerequisites before you start to upgrade. Generally, if you can run a prior version of XMB, you will be able to run XMB 1.9.2

Prerequisites

Also note that upgrading to XMB 1.9.2 is a one way process. If you need to back out, you will need to restore from backup.

Compatibility

Upgrades from XMB 1.6 and higher have been tested, including 1.9 beta. If you have earlier versions, this upgrade script may work, but it has not been tested.

Files used by the backup process:

  • upgrade.lib.php
  • upgrade.php
  • install.css
  • XMB_*.xmb (at the time of writing, XMB_1_9_2.xmb)

createFile.php creates the schema for XMB developers.

Known issues with upgrade

These issues are known. Most issues affect only a small percentage of users.

  • The upgrade can be quite resource-intensive, especially with large databases.
  • The script might time out or look to be inactive - do not stop it or close the browser window
  • The script can consume up to twice the amount of disk space you are currently using. Make sure you have enough room!
  • The upgrade script cannot cope with some indexes. If upgrade stops whilst trying to drop, alter or add an index, you will need to remove all indexes from the affected table. It's safe to remove as many indexes as possible as the upgrade script will put back the ones it needs. After dropping all additional indexes, simply restart the upgrade process.
  • The upgrade script occasionally cannot handle out of order columns, particularly in xmb_settings. If upgrade stops whilst performing a change to any table, you will need to change the table yourself. Once the table is closer to XMB's schema, just re-run upgrade.php

Instructions

WARNING: Hacks and data loss

!!!

This upgrade is designed to work when hacks are installed. However, all hack data including templates and code will be destroyed by the upgrade process! Proceed with extreme caution if you want to keep your hack data

!!!

Disable posting to your board in Admin -> Settings
 

Backups

Always take a backup before using *ANY* upgrade script! Although this is common sense, it's often forgotten.

0. Backup your database
1. Backup all your forum's files

If you have not performed a backup, XMB cannot help you if something goes wrong. You could lose all your data, posts, and users. You have been warned.

Do not proceed after this step if you have not performed a comprehensive backup.

2. Upload all of Nexus 1.9's files over the top of your existing forum files
3. Delete ./install/
4. Edit the new config.php to suit your needs
5. Upload the upgrade folder your forum
6. Run http://www.example.com/pathtoyourforum/upgrade/upgrade.php
7. When it succeeds, check that the upgrade folder has been deleted.
8. Log on your shiny new 1.9.2 board
9. Turn on the board in Admin -> Settings.

Adjust other new settings as you see fit.

Keeping XMB healthy

If your database is not looked after regularly, it can slow down. XMB 1.9.2 has many performance improvements which can be best utilized with a healthy database. XMB recommends administrators use Admin -> Analyze on their databases weekly, followed by Admin -> Optimize. This keeps your board in the best possible shape. Try it now!
 

Basic system administration

You should find out if your hoster performs regular backups for you. If they don't, you need to take backups manually. XMB 1.9.2 has a backup utility, but it is best if you use your hoster's utilities (such as phpMyAdmin or the mysqldump tool for this purpose).

Backups should be taken about once a week, and more often for very busy sites.

Security

All software has bugs, XMB included. You should check back with http://www.xmbforum.com/ regularly to see if you need to install patches.

When customizing your board, remember the following:

  • Changes to themes are supported
  • Modifications to XMB's PHP files, or to the database schema, or altering templates within the
    Admin -> Templates area makes it much harder for *YOU* to upgrade to the next service pack or a
    later version of XMB.
  • Try to keep the number of hacks down, and consider using "diff" to allow you to reapply them if you need
    to reload XMB from scratch.
